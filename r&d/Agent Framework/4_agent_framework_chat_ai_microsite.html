Provider roles

Provider routing is a model-profile concern, not controller-level branching and not multi-agent routing.

1

Default path

Azure OpenAI / OpenAI. Best Agent Framework parity, closest to current code, easiest Phase 1 migration.

2

Primary alternative

Anthropic Claude. Strong reasoning, long context, structured output, and useful provider diversity.

3

Secondary specialist

Gemini. Long-context, multimodal, embedding, cost/performance, and failover experiments.

4

RAG backbone

Existing HMC indexing + SmartSearch2 + Azure AI Search. Hosted provider search is optional augmentation.

HMC abstraction boundary

Use Agent Framework as the orchestration abstraction. Do not build a second agent framework inside HMC. ChatAI keeps Aivy instructions, tools, SignalR, persistence, user profile, document extraction, SmartSearch2 semantics, and policy. Provider-specific behavior belongs in one model/provider registry.

Suggested concepts
ProviderIdOpenAI, AzureOpenAI, Anthropic, Gemini.
ModelProfileProvider, model id, max context, tool/schema/vision/thinking capabilities.
ReasoningProfileDefault, Concise, Thinking, Deep. Mapped centrally to provider knobs.
AivyAgentFactoryBuilds the single Aivy agent for request-specific profile, tools, instructions, and run options.
AivyToolRegistryConverts HMC tools into Agent Framework AIFunctions.
AivyAgentRunServiceReplaces most of ChatJob orchestration while preserving HMC runtime semantics.
What survives

ChatContext survives because it is the SignalR/UI contract. ChatCoordinator survives because it owns stop, cancellation, and persistence coordination. SmartSearch2 survives because it is the HMC RAG backbone, not generic hosted search.

Migration path

Cull custom orchestration only after the side-by-side agent path proves parity.

Phase 0

Capture and feature flags

Add AgentFramework settings, provider profile settings, and decide approved procurement paths.

Phase 1

Single Aivy agent on OpenAI/Azure OpenAI

Build AivyAgentFactory, wrap existing WebSearch and SmartSearch2, preserve SignalR, stop, persistence, and Ask-Aivy semantics.

Phase 2

Tool and prompt cleanup

Convert JSON-string tools to typed methods, move status emission to middleware/event sinks, collapse duplicated loops.

Phase 3

Anthropic primary provider

Add provider path behind feature flag. Validate tool calling, structured output, streaming, SmartSearch2 behavior, and token accounting.

Phase 4

Gemini secondary provider

Use first for large-context, multimodal, embedding, cost/performance, and failover drills.

Phase 5

Memory / RAG revisit

Keep extraction, chunking, embedding, and Azure AI Search indexing. Consider context providers only after source blocks, filters, and retrieval quality survive.

Phase 6

Delete legacy orchestration

Delete only after parity and rollback window: accumulator, manual loops, compatibility layers, obsolete shims.

Risks

The dangerous failures are mostly semantic: losing UI block behavior, persistence semantics, or permission boundaries while chasing framework cleanliness.

Provider parity is uneven.Centralize ModelProfile capabilities and reject incompatible request options early.
SmartSearch2 loses block semantics.Make SmartSearch2 the Phase 1 parity gate. Keep a streaming tool event adapter.
Framework memory conflicts with HMC persistence.Keep Cosmos ChatHistory authoritative until a custom history provider proves parity.
Gemini bridge is not clean.Run it as a secondary-provider spike. Do not block OpenAI or Anthropic sequencing.
Long context tempts premature truncation removal.Keep token and Cosmos payload controls until load tests prove margins.
Hosted tools bypass HMC permissions.Keep HMC-owned tools for internal data. Hosted tools require separate review.

Immediate next move

Add AgentFramework:Enabled=false, define provider profiles, spike AivyAgentFactory on OpenAI/Azure OpenAI only, adapt WebSearchFunction and SmartSearch2Function, and capture parity gaps before deleting legacy orchestration.
